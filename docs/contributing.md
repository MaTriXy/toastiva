## Commit convention

Toastiva uses [Conventional Commits](https://www.conventionalcommits.org/) to drive automatic versioning and GitHub releases via [release-please](https://github.com/googleapis/release-please).

Every commit on `main` is parsed. The release bot opens (or updates) a single **Release PR** that bumps the version, regenerates `CHANGELOG.md`, and tags the release. Merging that PR publishes to npm.

> [!NOTE]
> While the version is `0.x`, breaking changes bump the **minor** by default (configured via `bump-minor-pre-major`). A real `1.0.0` only happens with an explicit `feat!:` once you decide to graduate.

## Scopes

Optional &mdash; useful for clarity but not required:

```
feat(provider): expose ref to toaster
fix(android): handle filter blur on Paper
```

## Release flow

1. Push commits to `main` (or merge a PR with a Conventional Commit title &mdash; squash merges work).
2. The **Release** workflow opens or updates `chore: release toastiva X.Y.Z`.
3. Review the autogenerated changelog and version bump in that PR.
4. Merge it &mdash; the workflow tags `vX.Y.Z` and creates the GitHub release.
5. Pull `main`, then publish to npm manually:

```bash
git pull
pnpm --filter toastiva publish --access public
```

> [!NOTE]
> Publishing is intentionally manual. CI handles versioning, changelog, tagging, and the GitHub release &mdash; only the npm push is left to you, so no `NPM_TOKEN` secret is required.

## Local checks

Match what CI runs:

```bash
pnpm --filter toastiva lint
pnpm --filter toastiva typecheck
pnpm --filter toastiva build
pnpm --filter toastiva pack:dry-run
```
